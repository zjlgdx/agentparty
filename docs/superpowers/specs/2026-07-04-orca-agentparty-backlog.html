P0 Issue A: Agent Identity / Context Card

Problem

#3 proved that identity/config drift breaks mentions-only, status attribution, and history auditability. CLI isolation can stop the local cause, but the web/channel surface still lacks a stable, visible context object for each active agent.

Why Now

After the config-isolation release, AgentParty can distinguish same-machine/same-repo agents. The product should make that distinction inspectable so humans and dispatchers do not have to infer identity from whoami, config paths, or message order.

Dependency

Required prior work#3 config/identity isolation accepted; use seq 65/67 style harness evidence as regression proof.
Related surfacesCLI whoami, status/presence, channel history, join-pack isolated config.

SignalConcrete check

Visible contextContext card shows config source/fingerprint, channel, workspace/worktree label, and current status.
Stable concurrent sessionsTwo same-directory A/B agents remain distinguishable in UI/history without manual whoami.

Issue Body Draft

Title: P0: Agent identity/context card for auditable multi-agent sessions

Why now:
#3 showed that identity/config drift breaks mentions-only routing, status attribution, and history auditability. The CLI isolation fix prevents the local drift, but humans still need a visible context surface in the channel UI/history.

Scope:
- Show each agent's identity, role/kind, and owner/account when available.
- Show config source/fingerprint, channel, workspace/worktree label, and current status.
- Make the same metadata visible from message history or presence.

Acceptance:
- Two same-directory A/B agents remain visually distinguishable in UI/history.
- A final synthesis/status message can be traced to the exact identity/context that posted it.
- No manual party whoami/config inspection is required for normal dispatcher review.

Depends on:
- #3 config/identity isolation release and harness evidence.

P0 Issue B: Completion Artifact / Final Synthesis Card

Problem

#1 defines the collaboration lifecycle: kickoff, replies or timeout, final synthesis, then done status. Today that lifecycle still appears as ordinary chat history, so a dispatcher has to manually scan messages to find the actual outcome.

Why Now

AgentParty is useful only if the shared channel retains the conclusion. A visible completion artifact turns the etiquette rule into a product affordance and makes review/replay cheap.

Dependency

Required prior work#1 final-synthesis-before-done protocol accepted.
Related surfacesreply_to, message kind/status, history filtering, done status note with summary_seq.

Acceptance Signals

SignalConcrete check
Artifact markerFinal synthesis is marked as an artifact in UI/API, not only a regular message.
Thread linkageArtifact shows kickoff/reply linkage via reply_to and summary_seq.
Timeout explicitnessArtifact can represent zero replies or timeout without skipping final synthesis.
Done orderingDone status cannot be treated as complete unless it references an existing final synthesis seq.

Issue Body Draft

Title: P0: Completion artifact card for final synthesis

Why now:
#1 establishes that brainstorm/review/dispatch work must close the loop in the channel. The final synthesis is currently just another chat line, which makes the shared outcome hard to find and review.

Scope:
- Mark final synthesis messages as completion artifacts in UI/API.
- Show kickoff seq, reply-to linkage, summary_seq, replies count, timeout flag, and related issue/PR links when provided.
- Surface the artifact from channel history without requiring manual scanning.

Acceptance:
- A completed brainstorm has a visible final synthesis artifact.
- The artifact can represent replies=0 and timeout=true.
- Done status references the artifact seq and appears after the artifact.
- History can filter or jump to completion artifacts.

Depends on:
- #1 completion protocol and status-done ordering.
