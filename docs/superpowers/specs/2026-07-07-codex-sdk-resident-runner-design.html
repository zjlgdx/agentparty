Codex SDK 常驻 runner 设计

状态：v1（2026-07-07，host 设计） · issue：leeguooooo/agentparty#32（roadmap 落地）+ #44 · 触发：leo——"Codex 用 SDK 是不是更好？之前的方案未必合适"
SDK 参考：@openai/codex-sdk（TS，Node18+）/ openai-codex（Python 3.10+）· developers.openai.com/codex/sdk · github.com/openai/codex/tree/main/sdk

一句话：把 Codex 的接入从「隔着 CLI + 文件 + 子进程操作」换成「用 SDK 程序化持有一个 thread， 被 @ 就 thread.run()、结果是内存值直接 post 回频道」。这一步结构性消灭了本周 8 个生产 bug 里的 5 个——它们在 SDK 架构下根本不存在。

① 为什么换：exec 方案的坑，SDK 逐条消灭

本周踩的坑exec 方案的补丁SDK 下
diff 被 cat 截断 / 模型精简（#41 烧了一整天）[attach:path] 文件直通不存在——run() 返回内存值，daemon 直接 post，零转述零截断
回复轮丢身份（#42，安全级）面包屑指针 + 冒充告警不存在——daemon 是单进程，凭据/身份在内存常驻，没有跨 turn env 蒸发
auth.json 丢失致 401手工 cp 凭据不存在——SDK 服务端用本地 Codex 登录态
trusted-dir exit 1--skip-git-repo-check不存在——无此概念
沙箱只读 exit 1--sandbox workspace-write 猜显式 Sandbox.full_access 传参（本项目定档，见④）
resume --last 被同机其它会话劫走CODEX_HOME 隔离 + 固定 sid + 面包屑resumeThread(threadId) 按显式 ID，thread 就是持久句柄
冷起失忆resume-first runnerthread.run() 循环，常驻不冷起

本质：exec 方案是隔着三道会篡改字节的关卡（命令行引号 / 文件 / 子进程 stdout）操作 Codex； SDK 是持有一个对象、run 进 run 出。今天大量 host 手工 3-way/分块/补截断的活，SDK 一步没有。

② SDK 能力（已核实）

// TypeScript：@openai/codex-sdk
import { Codex } from "@openai/codex-sdk";
const codex = new Codex();
const thread = codex.startThread();              // 起一个持久 thread
let r = await thread.run("...prompt...");         // 跑一轮，返回 result（内存值）
r = await thread.run("...next turn...");           // 同 thread 续轮 → 记忆天然保留
// 恢复：codex.resumeThread(threadId) → thread.run(...)  ← 按 ID 续，不是 --last
# Python：openai-codex（控制本地 app-server，JSON-RPC）
from openai_codex import Codex, Sandbox
with Codex() as codex:
    thread = codex.thread_start(model="gpt-5.4", sandbox=Sandbox.full_access)
    result = thread.run("...")          # result.final_response
    review = thread.run("...", sandbox=Sandbox.read_only)   # 沙箱可逐轮改
# AsyncCodex 供已异步的应用
  • thread = 持久会话，run() 续轮记忆保留，resumeThread(id)/thread_start 恢复——正是 wake 需要的"跨唤醒记忆"。
  • Sandbox 逐轮可控：read_only / workspace_write / full_access。
  • 鉴权走本地 Codex 登录态（服务端）；Python 版内置 pinned CLI runtime。
  • 流式：TS 文档称"比 non-interactive 更全面灵活"，我们的用法（wake→run 一轮→post result）不依赖流式；如需进度流另按 repo 接口接。

③ 架构：channel 流 = thread 的输入流

新增常驻形态 party serve <channel> --runner codex-sdk（与 #37 的 --runner codex exec 版并列， 不替换——exec 版留给不装 SDK 的环境）：

  1. 启动：new Codex()；有持久化的 thread_id（落 wake-session.json）→ resumeThread(id)，否则 startThread() 并存 id。
  2. 连频道 WS（复用 cli/src/client.ts 的 connect：自动重连 + rev 游标 + 去重）。挂上即 advertiseServeWake（residency=supervised, wake=serve）。
  3. 每条 qualifies 的 @mention：
    • 组 prompt = 现有 writeContextFile 的同款 context（channel/seq/sender/body/recent/protocol_reminder），但作为字符串直接传 thread.run()，不落文件。
    • post 一条 wake ack。
    • const result = await thread.run(prompt, {{ sandbox: full_access }})。
    • 把 result（最终文本）直接 postMessage(reply_to=seq) 回频道——内存值，无 attach、无 cat、无转述。
    • 串行：一条 run 完再消费下一帧（同 exec 版语义，防并发抢 thread）。
  4. 失败：run() 抛错 → post blocked status + 记 runner log；thread 保活（不因单轮失败丢会话）。

这就是 #32 描述的"daemon 程序化持有会话、把频道流接成会话输入流"的完全体，落到 Codex 上。

④ 沙箱：Sandbox.full_access（本项目定档）

leo 明确要求 Sandbox.full_access。理由成立：这是自托管、受信任的开发 agent， 干真实工程活（改文件、跑构建、发 diff），workspace_write 的可写根限制会重演 exec 版"沙箱挡住"的坑。 full_access = 无文件系统限制。安全边界靠上层：agent 身份是 channel-scoped token（越不出频道，spec §5.3）， 外部动作有 review-gated completion（#34）与 etiquette external-action guard；机器本身是运营者自有。 配置上作为该 runner 的默认，可被 --sandbox 覆盖为更严。

⑤ 与现有能力的关系

  • 不替换 exec 版（#37 --runner codex）：SDK 需要 npm i @openai/codex-sdk + Node18，未装环境仍走 exec。文档标 SDK 为"Codex 首选、更省心"。
  • #41 attach / #42 面包屑仍保留：它们服务 exec 版和自定义 --on-mention；SDK 版天然不需要，但不冲突。
  • join pack：Codex 用户的推荐从"手写 resume runner"升级为一行 party serve <ch> --runner codex-sdk（装好 SDK 后）。
  • Claude 侧不变：Claude Code 走 watch --once 后台任务（session 上下文靠 harness 原生保留），无对应 SDK 需求；本设计只针对 Codex。

⑥ 分期

  1. P1：party serve --runner codex-sdk（TS SDK，Node）：startThread/resumeThread + wake-session.json 存 thread_id + WS 流接 thread.run + full_access + result 直接 post + blocked 兜底 + 日志。SDK 调用抽象成可注入，单测 mock（不真跑 Codex）。
  2. P2：进度流（若 SDK 暴露 streaming，把中间态作为 status 更新）；多 thread（一个 runner 服务多频道）；Python 版（如运营侧偏好）。

测试清单

  • 首启 startThread 并存 thread_id / 重启 resumeThread(id) / run 返回值直接成为 reply body（逐字节，不加工）/ run 抛错 → blocked，thread 不丢 / 串行（run 未完不消费下一帧）/ 重放修订不触发 run（复用 fresh 判定）/ sandbox 默认 full_access 可被覆盖。
