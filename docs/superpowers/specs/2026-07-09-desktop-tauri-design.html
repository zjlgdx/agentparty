1. 目标与非目标

目标:一个 macOS/Windows/Linux 桌面应用,复用现有 web/ 前端全部代码,连接远端 worker(prod agentparty.leeguoo.com / xdream agentparty.pwtk-dev.work 可切换),并提供网页给不了的桌面能力:系统通知、托盘常驻、未读角标。

非目标:不 fork 一套桌面版 UI(web 代码是唯一 UI 源);不内置 worker(服务端仍在云上);M1/M2 不做本地 agent 进程管理(留 M3)。

2. 现状事实(这四条约束决定了整个方案)

#事实出处对桌面的影响
F1web 的所有网络调用是相对路径:30 处 fetch("/api/...") 集中在 web/src/lib/api.ts,2 处在 oidc.ts;WebSocket 用 location.host 拼 URLweb/src/lib/api.ts、oidc.ts:62,164、ws.ts:55-62桌面 webview 的 origin 是 tauri://localhost,相对路径全部打空 → 必须引入可配置 API base(改动小,调用点集中)
F2worker 没有任何 CORS 处理(grep 全仓无 access-control)worker/src/index.ts桌面壳跨域调 /api/* 会被 webview 的 CORS 拦 → worker 加白名单 CORS 中间件(D3)
F3会话 token 存 localStorage(lib/api.ts:26,54);web 已内置粘贴 token 登录(App.tsx:388);OAuth 登录 redirect_uri = window.location.origin + "/auth/callback"web/src/lib/api.ts、App.tsx、oidc.ts:134OAuth 授权页不会跳回 tauri:// → M1 用现成的粘贴登录,M2 做配对码登录(D4)
F4web 已有 @mention 浏览器通知开关(NotifyToggle.tsx,标准 Notification API + localStorage opt-in)web/src/components/NotifyToggle.tsx桌面通知不用重写业务逻辑,把 Notification 桥到系统通知即可(M2)

3. 关键决策记录

D1 壳选 Tauri v2,不用 Electron。理由:① 产物 ~10MB vs ~150MB;② Rust 壳与 screenpipe 同栈,团队有现成经验;③ Tauri v2 插件生态已覆盖我们要的全部能力(notification / tray / deep-link / single-instance / updater);④ UI 全在 webview 里,Electron 的 Node 集成优势用不上。
D2 monorepo 加 desktop/ workspace,不开新仓库。桌面构建直接消费 ../web/dist,web 一次构建两端使用;版本随主仓 v* tag 走。
D3 跨域方案:worker 加"桌面 origin 白名单" CORS 中间件,只放行 tauri://localhost / http://tauri.localhost(Windows 上 Tauri 的 origin)。不用 Tauri 的 http 插件绕 CORS——那要求把 web 的 fetch 全换成插件 API,侵入大。安全性:浏览器网页无法伪造 tauri:// origin,白名单不会给 web 攻击面开口;Authorization 头本来就是显式带的,预检放行即可。WebSocket 不受 CORS 约束,不用处理。
D4 登录分两步走:M1 粘贴 token(现成功能,零开发);M2 配对码(provider 无关)。不做"桌面内嵌 OAuth"——Lark/OIDC 的 redirect_uri 白名单都是 https 页面,桌面拿 code 要么占端口起 loopback(Lark 后台不一定让配 localhost),要么劫持回调域,都脏。配对码模式(手机扫码登录同构)干净且对 prod(OIDC)/xdream(Lark)一视同仁:已登录的网页生成一次性短码 → 桌面拿短码换自己的 session token。
D5 API base 是运行时可配,不是编译期常量。桌面设置页可切 prod/xdream;默认值由构建环境变量 VITE_API_BASE 提供。浏览器版 base 恒为空串,行为与今天完全一致(零回归)。

4. M1 最小可用桌面壳

4.1 web 改造:引入 apiBase(桌面/浏览器共用)

新文件 web/src/lib/base.ts:

// API base:浏览器版恒为 ""(相对路径,行为不变);桌面版由设置页写 localStorage,
// 默认取构建期 VITE_API_BASE。切换环境 = 改 localStorage + 整页 reload。
const BASE_KEY = "ap_api_base";

export function apiBase(): string {
  try {
    return localStorage.getItem(BASE_KEY) ?? (import.meta.env.VITE_API_BASE as string | undefined) ?? "";
  } catch {
    return "";
  }
}

export function setApiBase(base: string): void {
  localStorage.setItem(BASE_KEY, base.replace(/\/+$/, ""));
}

export function apiUrl(path: string): string {
  return apiBase() + path;
}

// ws.ts 用:base 为空沿用 location(现状);否则从 base 推导 wss host
export function wsUrl(path: string): string {
  const base = apiBase();
  if (base === "") {
    const proto = location.protocol === "https:" ? "wss" : "ws";
    return `${proto}://${location.host}${path}`;
  }
  return base.replace(/^http/, "ws") + path;
}

改造点(机械替换,一次提交):

文件改法
web/src/lib/api.ts(30 处)fetch("/api/x") → fetch(apiUrl("/api/x"));模板串同理
web/src/lib/oidc.ts(:62、:164 两处)同上。redirect_uri 的 window.location.origin 不动——OAuth 只发生在浏览器里(D4)
web/src/lib/ws.ts(:55-62)URL 拼接换 wsUrl(...)
web/src/state.test.ts 等现有测试base 默认 "",应零改动;新增 base.ts 单测:空 base 相对路径、带 base 拼接、ws 推导

4.2 worker 改造:桌面 CORS 中间件

worker/src/index.ts,挂在所有 /api 路由之前:

// 桌面壳(Tauri webview)的跨域放行:浏览器网页伪造不了这两个 origin,
// 放行不会扩大 web 攻击面。凭据仍是显式 Authorization 头,不涉 cookie。
const DESKTOP_ORIGINS = new Set(["tauri://localhost", "http://tauri.localhost", "https://tauri.localhost"]);

app.use("/api/*", async (c, next) => {
  const origin = c.req.header("origin") ?? "";
  if (!DESKTOP_ORIGINS.has(origin)) return next();
  if (c.req.method === "OPTIONS") {
    return new Response(null, {
      status: 204,
      headers: {
        "access-control-allow-origin": origin,
        "access-control-allow-methods": "GET,POST,PUT,DELETE,OPTIONS",
        "access-control-allow-headers": "authorization,content-type",
        "access-control-max-age": "86400",
      },
    });
  }
  await next();
  c.res.headers.set("access-control-allow-origin", origin);
});

配套 vitest:OPTIONS 预检 204 带头 / 白名单外 origin 无 CORS 头 / 白名单内 GET 响应带 allow-origin。

4.4 设置页(web 内小组件,仅桌面态显示)

判定桌面态:location.protocol === "tauri:" 或存在 window.__TAURI_INTERNALS__(封装成 web/src/lib/desktop.ts 的 isDesktop())。设置面板提供:环境切换(prod/xdream 两个预设 + 自定义 URL,调 setApiBase 后 reload)、当前登录身份、退出登录。挂在现有顶栏菜单里,浏览器版不渲染。

4.5 M1 登录(零开发)

用 web 现成的粘贴 token 登录(App.tsx:388)。token 从哪来,给用户三条路(写进设置页提示文案):① 已登录的浏览器 web 端 localStorage(M1.5 在 web 设置里加"复制访问令牌"按钮);② CLI:party login 后 cat ~/.agentparty/config.json 里的 token;③ 管理员 party token 铸的 ap_ token。

4.6 M1 验收标准

  1. cd desktop && bun run dev 起窗口,连本地 wrangler dev 全功能可用;

5. M2 桌面体验

5.1 系统通知(复用 NotifyToggle 业务逻辑)

装 tauri-plugin-notification(Cargo + JS 包 @tauri-apps/plugin-notification),capabilities 加 "notification:default"。改造 NotifyToggle.tsx:抽一个 notify(title, body) 适配层——isDesktop() 时调插件 API(权限请求 + 发送),否则走现有 Notification API。点击通知聚焦窗口并跳对应频道(插件的 action 事件里调 getCurrentWindow().setFocus())。

托盘:TrayIconBuilder(main.rs ~30 行):左键显示/隐藏窗口,菜单含"打开 / 检查更新 / 退出";关窗改为隐藏(拦 WindowEvent::CloseRequested),真正退出走托盘菜单。角标:web 侧已有未读计数逻辑,新增一个桥:未读数变化时调 Tauri command set_badge(n)(macOS 用 set_badge_count,Windows 用 overlay icon)。

5.3 配对码登录(worker + web + desktop 三端各一小块)

D1 migration worker/migrations/0018_desktop_pairing.sql:

-- 桌面配对码:已登录网页签发,桌面一次性兑换成自己的 session token
CREATE TABLE desktop_pair_codes (
  code       TEXT PRIMARY KEY,   -- 20 字符 base32 高熵(crypto.getRandomValues)
  account    TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  expires_at INTEGER NOT NULL,   -- created_at + 5 分钟
  claimed_at INTEGER             -- 非 NULL = 已用,拒绝二次兑换
);
端改动
workerPOST /api/desktop/pair/new(requireBearer,human 账号会话)→ 生成 code 落表,返回 {code, expires_in:300};POST /api/desktop/pair/claim(无鉴权,body {code})→ 校验未过期未使用 → 标记 claimed → 用该 account 走 upsertHumanSessionToken 同款逻辑铸一枚新的桌面会话 token 返回。限流:同 IP claim 失败 10 次/小时封禁(D1 计数即可)。过期行由下次 new 时顺手清理
web设置里加"连接桌面应用":调 pair/new,展示 6 秒大字 code + agentparty://pair?code=...&host=... 链接(点击尝试唤起桌面应用)
desktop装 tauri-plugin-deep-link + tauri-plugin-single-instance(deep link 的前置),注册 agentparty:// scheme;收到 pair 链接或手输 code → 调 claim → setApiBase(host) + 存 token + reload。登录页加"输入配对码"输入框(不依赖 deep link 也能用)

5.4 M2 验收标准

  1. 桌面在后台时被 @,系统通知弹出,点击跳到对应频道;
  2. 关窗后托盘常驻,WS 保持,未读角标随消息增长;
  3. 浏览器点"连接桌面应用"或手输配对码,桌面 10 秒内完成登录;配对码只能用一次、5 分钟过期;
  4. bun run check 全绿。

8. 发布与 CI

沿用仓库惯例:push main 只跑 CI,发布靠 v* tag;产物走 GitHub Release 附件分发(不进 npm/商店)。新增 workflow 用官方 tauri-apps/tauri-action:

# .github/workflows/desktop-release.yml(骨架)
on:
  push:
    tags: ["v*"]
jobs:
  desktop:
    strategy:
      matrix:
        include:
          - platform: macos-latest        # 同时出 aarch64 + x86_64,用 args 控制
          - platform: windows-latest
          - platform: ubuntu-22.04
    runs-on: ${{ matrix.platform }}
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v2
      - uses: dtolnay/rust-toolchain@stable
      - run: bun install
      - run: cd web && VITE_API_BASE=https://agentparty.leeguoo.com bunx vite build
      - uses: tauri-apps/tauri-action@v0
        env: { GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }} }
        with:
          projectPath: desktop
          tagName: ${{ github.ref_name }}
          releaseName: "AgentParty ${{ github.ref_name }}"

macOS 签名/公证初期跳过(用户右键打开或 xattr -dr com.apple.quarantine,README 写明);拿到开发者证书后再加 APPLE_CERTIFICATE 系列 secret,workflow 不用改结构。

10. 风险与坑

坑应对
Windows 上 Tauri origin 是 http://tauri.localhost,mac 是 tauri://localhostCORS 白名单两个都放(§4.2 已含);web 侧 isDesktop() 不要用 protocol 判断唯一依据,优先 __TAURI_INTERNALS__
webview 里 localStorage 存 token,清缓存即掉登录可接受(与浏览器一致);M3 如需更稳换 tauri store 插件,适配层已隔离
vite dev 的 /api 代理只在 dev 生效,打包后忘配 base → 白屏无数据base 为空且 isDesktop() 时,登录页直接引导先选环境(设置页首启弹出)
未签名 macOS 包被 Gatekeeper 拦§8:README 写右键打开;后续补证书
deep link 在 Linux 需要 .desktop 文件注册,行为各发行版不一配对码手输路径是第一公民,deep link 只是加速;不给 Linux 深链做保证
CORS 放行后有人担心安全D3 已论证:origin 伪造不可行、无 cookie 凭据;评审时引用该决策

11. 小白上手指南(从零到 M1 窗口跑起来)

  1. 装环境:curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh(Rust);装 bun;macOS 需要 xcode-select --install。
  2. 基线:仓库根 bun install && bun run check 确认全绿。
  3. web 改造先行(§4.1):建 base.ts → 机械替换三个文件 → cd web && bun test && bunx vite build。这步不碰桌面,单独一个 commit,浏览器零回归是它的验收。
  4. worker CORS(§4.2):中间件 + vitest,单独 commit。
  5. 建壳:mkdir desktop && cd desktop,按 §4.3 逐文件创建(或 bunx @tauri-apps/cli init 后对照 §4.3 改配置);bunx tauri icon path/to/1024.png 生成图标。
  6. 本地跑:终端 A cd worker && bun run dev(wrangler dev :8787);终端 B cd desktop && bun run dev——窗口出现、走 vite 代理连本地 worker。
  7. 打包连远端:cd web && VITE_API_BASE=https://agentparty.pwtk-dev.work bunx vite build && cd ../desktop && bunx tauri build;装 dmg,粘贴 token 登录,过 §4.6 验收。
  8. 提交:中文 conventional commits,PR 关联 #63。M2 从 §5.1 通知开始做,每个小节一个 PR。

工作量参考:M1 约 1~2 人日(大头是 web 改造与验证),M2 约 3~4 人日(配对登录占一半),CI 约 0.5 人日。
