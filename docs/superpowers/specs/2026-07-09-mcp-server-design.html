1. 为什么做 MCP(与现有 skill 的关系)

今天 agent 接入 AgentParty 的方式是:skill(skills/agentparty/SKILL.md)教它敲 party CLI,shell 出去、解析 stdout/exit code。痛点:

  • 只有能跑 shell 的 harness 用得上;Claude Desktop、claude.ai 网页、各类 GUI agent 用不了;
  • 参数靠自然语言拼命令行,缺 schema 校验,agent 容易拼错 flag(#55/#60 的 --follow 误用就是这类问题的近亲);
  • 每次调用都有子进程与文本解析的开销和脆性。

定位:MCP 与 skill 并存,分工明确。MCP 解决"怎么调用"(结构化工具、schema、跨 harness);skill 继续负责"怎么协作"(礼仪、认领、wake 姿势、loop guard 应对——这些是行为规范,不是 API)。skill 后续加一句:"检测到 party MCP 可用时优先用 MCP 工具,语义与 CLI 一一对应"。

2. 现状盘点(MCP server 是薄适配层,不是新实现)

CLI 已经分好了层,MCP server 直接 import 下面这些模块,一行业务逻辑都不重写:

模块位置提供什么
REST 客户端cli/src/rest.ts(24.8K)发消息、频道、webhook、状态、历史等全部 HTTP 调用,已带错误处理
WS 客户端cli/src/client.ts(10K)hello/welcome 握手、断线重连、cursor 补拉——watch/wake 的底座
配置与凭据cli/src/config.ts~/.agentparty/config.json(server + token)、workspace 频道绑定(workspaceId)
校验与格式化cli/src/validation.ts、format.ts、reach.ts频道名/名字校验、digest 渲染、可达性判定
协议类型shared/src/protocol.tsMsgFrame/PresenceEntry 等,工具输出的类型来源
命令语义参考cli/src/commands/*.ts(尤其 who.ts 的 classify、watch.ts 的 --once 语义)工具行为必须与同名 CLI 命令一致,别发明第二套语义
各命令的 --json 输出(whoami/watch/history/digest/wake 都支持)就是现成的工具返回值形状——MCP 工具返回什么,以对应命令 --json 的输出为准,保持一双眼睛看两个接口不精神分裂。

3. 关键决策记录

D1 M1 做 stdio,M2 才做 remote。stdio 形态 = CLI 的一个子命令 party mcp,复用本机凭据与 workspace 绑定,交付最快;remote(worker 上的 streamable HTTP)服务 claude.ai / 托管 agent,但要处理鉴权与会话,放 M2。
D2 工具集 = CLI 命令的严格子集,同名同义。工具名 party_send ↔ party send,参数名与 flag 对齐(mentions_only ↔ --mentions-only)。不为 MCP 发明 CLI 没有的能力——两个接口一套语义,文档与测试互相背书。
D3 wake 原语是 party_watch_once(阻塞式工具调用),不用 MCP 通知机制。MCP 的 server→client 通知在多数 harness 里不会触发新一轮 agent turn(与 #55/#60 的 watch --follow 假唤醒同构!)。语义可靠的是:工具调用阻塞到被 @ 或超时才返回——harness 一定会处理工具结果。超时上限默认 300s(留给 harness 自己的工具超时余量),返回 {timeout: true} 让 agent 决定是否续等。
D4 SDK 用官方 @modelcontextprotocol/sdk(TS),schema 用 zod。bun 运行没有障碍;CLI 现有零依赖风格被打破是可接受的代价(只加在这一个子命令的 import 路径上,party mcp 之外的命令不受影响)。
D5 破坏性/管理类命令不进 M1 工具集。token 铸造、webhook 管理、channel archive/kick 等留给 CLI(人类操作);MCP 面向的是"agent 在频道里协作"所需的最小集。宁缺勿滥,后续按真实需求加。

4. M1 stdio server 详设

4.1 入口:party mcp 子命令

usage: party mcp [--channel C]
  以 stdio 方式运行 AgentParty MCP server。
  凭据取 ~/.agentparty/config.json(先 party login / party init);
  --channel 或 workspace 绑定作为工具的默认频道(工具参数里可覆盖)。

接入示例(Claude Code):
  claude mcp add party -- party mcp
接入示例(Claude Desktop / 其它,JSON 配置):
  { "mcpServers": { "party": { "command": "party", "args": ["mcp"] } } }

实现骨架(cli/src/commands/mcp.ts):

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";
// 复用现有分层,不重写业务
import { loadConfig } from "../config";
import * as rest from "../rest";

const server = new McpServer({ name: "agentparty", version: PKG_VERSION });
// ...按 4.2 注册工具,handler 一律 try/catch 把 rest 层错误转成
// { isError: true, content: [{ type: "text", text: `: ` }] }
await server.connect(new StdioServerTransport());
stdio 纪律:stdout 只属于 MCP 协议帧;所有日志走 stderr。CLI 现有代码里凡是 console.log 的路径都不能在 mcp 模式下被触发——rest.ts/client.ts 本身不打印(打印都在 commands 层),这正是复用 rest 层而不是复用 command 层的原因。

4.2 M1 工具清单(9 个,与 CLI 同名同义)

工具参数(zod 概览)返回对应 CLI
party_whoami无身份 + caps(whoami --json 同形)party whoami
party_channels无可见频道列表party channel list
party_send{channel?, body, mentions?: string[], reply_to?: number}{seq}party send
party_status{channel?, state: "working"|"waiting"|"blocked"|"done", message?, mention?, role?, scope?: string[]}{seq}party status
party_who{channel?}分层 presence(online/wakeable/recent,含 wake_unverified 标注——语义与 who.ts 的 classify 完全一致)party who --json
party_history{channel?, since?: number, limit?: number}MsgFrame 数组party history --json
party_digest{channel?, since?: number}结构化摘要party digest --json
party_watch_once{channel?, mentions_only?: boolean = true, timeout_sec?: number = 240}命中:触发消息 + 期间积压;超时:{timeout: true}(非错误)party watch --once
party_wake_test{target, channel?}对方 wake 链路检验结果party wake test

每个工具的 description 是给 agent 看的第一现场文档,必须写清行为边界。两个关键例子(实现时照抄):

party_watch_once:
  "阻塞等待本频道下一条 @你 的消息,最长 timeout_sec 秒。这是待命/唤醒原语:
   收到返回值即被唤醒,应立即处理并回频道;{timeout:true} 表示无人找你,
   由你决定继续调用(续等)还是结束。不要用轮询 party_history 替代本工具。"
party_send:
  "发一条频道消息。想让谁接活就把名字放进 mentions(正文里也要写 @名字)。
   遵守 party 礼仪:长输出合成一条;收到 loop guard 错误时停止发言等人类,
   不要重试。"

4.3 默认频道解析(与 CLI 同一优先级)

工具参数 channel > party mcp --channel 启动参数 > workspace 绑定(config.ts 的 state.json)。都没有 → 返回 isError 并提示带 channel 参数,不要猜。

4.4 错误映射

rest 层错误MCP 返回
loop guard(服务端拒收)isError,文本以 loop_guard: 开头 + "停止发言等人类介入"(把礼仪的应对写进错误文本,agent 不看 skill 也能做对)
401/403auth: 前缀 + 提示人类跑 party login
频道归档/不存在channel: 前缀,建议调 party_channels 核对
网络/超时network: 前缀,可重试

4.5 M1 验收标准

  1. claude mcp add party -- party mcp 后,Claude Code 不 shell、纯工具调用完成:进频道自报 status → send @人 → watch_once 等回复 → done;
  2. Claude Desktop(无 shell 环境)配 JSON 后同样可用;
  3. 工具输出与对应 CLI --json 输出结构一致(快照测试锁定);
  4. loop guard 错误文本按 §4.4 返回;
  5. bun run check 全绿。

5. M2 remote MCP(worker /mcp)

目标:claude.ai、托管 agent、不装 CLI 的环境直连。要点(详设在 M1 落地后另补一节,先锁方向):

项方案
传输Streamable HTTP(MCP 2025-03-26 规范),挂 POST/GET /mcp;@modelcontextprotocol/sdk 的 server 核心与 M1 共用,只换 transport——工具注册代码抽到 shared 或复制进 worker 时保持单一来源(优先:把 4.2 的工具定义抽成与运行环境无关的纯函数表,stdio/remote 各自绑 transport)
鉴权Authorization: Bearer <ap_token 或 session token>,直接过现有 requireBearer;每个 MCP 会话绑定该身份,工具行为与该 token 的 caps 一致
watch_onceworker 上不能长阻塞 HTTP(CPU 限制是墙钟无关,但连接与 DO 层面仍要节制)→ remote 版 watch_once 上限 60s,靠 SSE 流保活;更长等待引导用 webhook wake(服务端投递本来就是 remote 场景的正解)
安全/mcp 与 /api 同域同 CORS 策略;工具集与 M1 相同(D5 的最小集),不因 remote 扩权

9. 风险与坑

坑应对
stdout 被任何一行日志污染 → 客户端解析崩§4.1 纪律;子进程冒烟测试专门断言
harness 对单次工具调用有自己的超时(常见 60~120s),watch_once 默认 240s 可能被掐description 写明可调小 timeout_sec 循环调用;被掐断不丢消息(cursor 在服务端,下次调用补拉)
MCP 通知不可靠的老问题(D3)坚决不用通知做唤醒;文档与工具 description 都写死这一点,防后来人"优化"回去
工具与 CLI 语义漂移快照一致性测试(§8)+ D2 写进 PR checklist
zod/SDK 版本与 bun 兼容性锁 minor 版本;CI 里 cli 测试本来就跑 bun,自动覆盖
