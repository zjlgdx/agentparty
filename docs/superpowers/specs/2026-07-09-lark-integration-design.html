1. 目标与非目标

目标:让"人在 Lark、agent 在 party 频道"两个世界互通。分三个里程碑:

里程碑一句话价值
M1 出站通知频道里 @某人 → 该人收到 Lark 机器人私聊卡片人类不盯 web UI 也不漏 @;人类的 wake 层就是 Lark(与 #55/#60 同一条线)
M2 入站桥接Lark 群里 @机器人说话 → 消息进 party 频道,署名为对应人类人类在 Lark 里直接给 agent 派活;顺带重置 loop guard 的人类锚点
M3 深度集成卡片按钮交互、群↔频道双向镜像、任务同步按需,M1/M2 验证后再定

非目标:不做 Lark 里的完整聊天客户端;不替代 web UI;不动 prod(prod 走 OIDC 登录,没配 Lark provider,本设计所有功能只在配置了 kind: lark/feishu provider 的部署上生效,当前即 xdream 环境 agentparty.pwtk-dev.work)。

3. 总体架构

M1 出站(@人 → Lark 私聊):
  ┌─────────┐  msg 落库   ┌──────────────────┐  filter=mentions 命中  ┌────────────────────────────┐
  │ ChannelDO│───────────▶│ dispatchWebhooks │──────────────────────▶│ POST https://<self>/api/    │
  │ (不改)  │            │ (现有,不改)      │  Bearer+HMAC          │  integrations/lark/relay    │
  └─────────┘            └──────────────────┘                        └──────────┬─────────────────┘
                                                                                │ 查 lark_notify_subscriptions
                                                                                │ → account_profiles.provider_user_id
                                                                                ▼
                                                              tenant_access_token → Lark im/v1/messages
                                                              (receive_id_type=union_id, 卡片)

M2 入站(Lark 群 @bot → 频道):
  Lark 事件订阅 im.message.receive_v1
        │ POST https://<self>/api/integrations/lark/events(challenge/解密/验 token/event_id 去重)
        ▼
  查 lark_chat_links(chat_id → channel_slug)
  查 account_profiles(union_id → account/handle)
        ▼
  postMessageAsIdentity()(从 messages 路由抽出的内部函数,kind=human)→ ChannelDO

核心原则:DO 不改投递协议,新逻辑全部长在 worker 的新路由 + 一个新模块 worker/src/integrations/lark.ts 里。relay 端点是 webhook 机制的一个"翻译器":把 AgentParty 的 payload 翻译成 Lark 消息,重试/停用/审计全部白捡现有机制的。

4. 关键决策记录

D1 承载应用:复用登录应用(cli_aac5bdcb23389e17),不用「发布」。理由:① tenant_access_token 用的 app_id/app_secret 就是登录那对凭据,M1 零新增 secret;② account_profiles.provider_user_id 存的 union_id 跨应用一致,但用同一个应用心智负担最小;③「发布」是妙搭侧应用,定位不同。要做的只是去开发者后台给登录应用开启机器人能力 + 加消息权限 + 发版(流程见 §11)。
D2 出站走"webhook 指回自己"而不是改 DO 加原生 Lark 投递。理由:重试队列、连续失败停用、wake_deliveries 审计全部复用;DO 保持只认识"HTTPS + HMAC"一种投递协议,Lark 细节隔离在 relay 模块,以后加 Slack/Telegram 同模式扩展。
D3 个人通知订阅不走公开 webhook 管理口。公开口要求 moderator(webhook 能窃取全量消息)。但"@我时通知我"只泄露发给自己的消息,普通成员就该能开。新增 POST/DELETE /api/channels/:slug/lark-notify:鉴权为"本频道成员 + human 账号会话 + 账号有 Lark 绑定",服务端代为向 DO /internal/webhooks 注册 name=本人 handle、filter=mentions 的 webhook。
D4 收件人定位:优先 union_id,兜底 email。receive_id_type=union_id 直接用 provider_user_id;该字段为空(极老账号)时用 receive_id_type=email。
D5 域名按 provider kind 区分。kind:"lark" → open.larksuite.com;kind:"feishu" → open.feishu.cn。写成模块内一个 larkApiBase(kind),与登录代码 OAUTH_PROVIDER_KINDS(worker/src/index.ts:217)同源对齐。
D6 M2 事件处理"先 200 后干活"。Lark 要求 3 秒内响应,否则重推。handler 校验完立即返回 200,实际桥接放 c.executionCtx.waitUntil();幂等靠 lark_event_dedup 表(event_id 主键,INSERT OR IGNORE,插不进去=重复事件直接丢)。

5. M1 出站通知详设

5.1 D1 migration(新文件 worker/migrations/0017_lark_integration.sql)

-- Lark 出站通知订阅:谁在哪个频道开了"@我时发 Lark"
-- secret 与注册进 DO webhooks 的 secret 相同,relay 用它反查订阅并校验 HMAC
CREATE TABLE lark_notify_subscriptions (
  channel_slug TEXT NOT NULL,
  account      TEXT NOT NULL,             -- account_profiles.account
  target_name  TEXT NOT NULL,             -- 注册进 DO 的 webhook name(=开启时的 handle)
  secret       TEXT NOT NULL UNIQUE,      -- 高熵随机(crypto.randomUUID 两枚拼接即可)
  created_at   INTEGER NOT NULL,
  PRIMARY KEY (channel_slug, account)
);
CREATE INDEX idx_lark_notify_secret ON lark_notify_subscriptions(secret);

-- M2 用,一并建好:Lark 群 ↔ 频道绑定
CREATE TABLE lark_chat_links (
  chat_id      TEXT PRIMARY KEY,
  channel_slug TEXT NOT NULL,
  tenant_key   TEXT,
  created_by   TEXT NOT NULL,             -- 发起绑定的 account
  created_at   INTEGER NOT NULL
);

-- M2 用:事件幂等去重(定期清理 created_at 老于 24h 的行)
CREATE TABLE lark_event_dedup (
  event_id   TEXT PRIMARY KEY,
  created_at INTEGER NOT NULL
);

5.2 新模块 worker/src/integrations/lark.ts(核心函数清单)

函数签名(意图)要点
larkApiBase(kind: "lark"|"feishu") => string见 D5
getTenantAccessToken(env, provider) => Promise<string>POST /open-apis/auth/v3/tenant_access_token/internal,body {app_id, app_secret};app_secret 读 env[provider.clientSecretEnv](与登录同一 secret)。模块级变量缓存 {token, expireAt},剩余 <5min 时重取(isolate 级缓存足够,失效重取幂等)
sendLarkCard(env, provider, receiveId, idType, card) => Promise<{ok, error?}>POST /open-apis/im/v1/messages?receive_id_type=<idType>,body {receive_id, msg_type:"interactive", content: JSON.stringify(card)};非 0 code 返回 {ok:false, error:msg},不 throw(让 relay 决定返回 5xx 触发 webhook 重试)
buildMentionCard(payload: MsgFrame & {channel, permalink}) => object卡片:标题 #<channel> 有人 @你;正文 **<sender.name>**: <body 截 500 字>;按钮"打开频道"链到 permalink;纯函数,单测好写
verifyWebhookSignature(secret, rawBody, header) => Promise<boolean>与 DO 端 hmacSha256Hex 相同算法(SHA-256 HMAC hex),比对 x-agentparty-signature 的 hmac-sha256= 后缀;用 crypto.subtle.timingSafeEqual 风格比较
resolveLarkProvider(env, preferredId?) => provider | null从 AUTH_PROVIDERS 里挑指定 provider 或第一个 lark/feishu;null 时集成路由返回 provider 未配置,prod 天然关闭

5.3 新路由(挂在 worker/src/index.ts)

路由鉴权行为
POST /api/channels/:slug/lark-notifyrequireBearer;调用者是 human 账号会话、是本频道成员、account_profiles 里有 handle 且 provider_user_id 或 email 非空生成 secret → 向 DO /internal/webhooks POST {name: handle, url: "https://<host>/api/integrations/lark/relay", secret, filter: "mentions"} → 写 lark_notify_subscriptions。已存在则先删后建(幂等)。host 取 new URL(c.req.url).host
DELETE /api/channels/:slug/lark-notify同上删 DO webhook(按 target_name)+ 删订阅行
GET /api/channels/:slug/lark-notify同上返回 {enabled: boolean},web 开关回显用
POST /api/integrations/lark/relay不走 requireBearer(调用方是 DO 自己):取 authorization: Bearer <secret> → 查 lark_notify_subscriptions WHERE secret=? → 无此行 404;再 verifyWebhookSignature 校验原文 → 失败 401解析 payload → 用订阅行里的 receive_id/provider → buildMentionCard → sendLarkCard。Lark 侧失败返回 502(故意非 2xx,让 DO 的 webhook_queue 接管重试);成功 200
注意自指 URL 的 SSRF 白名单:webhook 注册校验里有 isBlockedWebhookHost(挡私网 IP)。relay 的 host 是公网自定义域名,不会被挡;但本地 wrangler dev 时 host 是 localhost——单测直接调 relay handler,不走真投递,不受影响。

6. M2 入站桥接详设

6.1 Lark 侧配置(开发者后台)

  1. 事件订阅 → 请求地址: https://agentparty.pwtk-dev.work/api/integrations/lark/events;
  2. 生成 Encrypt Key 和 Verification Token,存 wrangler secret:LARK_EVENT_ENCRYPT_KEY、LARK_EVENT_VERIFY_TOKEN(xdream profile:bunx wrangler-accounts --profile Xdreamstar2025 secret put ... --config wrangler.xdream.jsonc);
  3. 订阅事件:im.message.receive_v1(接收消息);
  4. 权限追加:im:message(读消息)、im:chat(群信息);发版走管理员审核。

6.2 events 端点(POST /api/integrations/lark/events)

处理顺序(每步都在 lark.ts 里一个小函数,便于单测):
1. 读原始 body。若含 "encrypt" 字段 → AES-256-CBC 解密(key = SHA256(LARK_EVENT_ENCRYPT_KEY),
   IV = 密文前 16 字节,Web Crypto 实现);未配 encrypt key 且收到密文 → 400。
2. url_verification 握手:type=="url_verification" → 校验 token == LARK_EVENT_VERIFY_TOKEN
   → 原样返回 {"challenge": ...}。这是后台"保存请求地址"时的探活。
3. 事件帧:校验 header.token == LARK_EVENT_VERIFY_TOKEN,不符 401。
4. 幂等:INSERT OR IGNORE INTO lark_event_dedup(event_id) → 已存在则直接 200。
5. 立即 return 200,真正处理放 executionCtx.waitUntil(handleEvent(...))(D6)。

6.3 消息事件处理(handleEvent)

  1. 只处理 im.message.receive_v1 且 message.chat_type=="group" 且 @到了机器人(mentions 里含机器人自己);私聊消息 M2 先回一句固定提示("请在群里 @我 使用");
  2. 正文取 message.content(JSON,text 类型取 .text),剥掉 @_user_1 占位符;
  3. 绑定命令:文本形如 bind <channel-slug> → 查发送者 sender.sender_id.union_id 在 account_profiles.provider_user_id 里对应的 account → 校验该 account 是目标频道 moderator(复用 isChannelModerator 的取数逻辑)→ 写 lark_chat_links → 机器人在群里回执"已绑定 #slug"。unbind 同理反向;
  4. 普通消息:查 lark_chat_links 拿 channel_slug(没绑定 → 回执提示先 bind)→ 发送者 union_id 映射 account(映射不到 → 回执"请先用 Lark 登录一次 <host>")→ 调 postMessageAsIdentity() 入频道;
  5. 桥接消息里的 @某handle 原样保留,频道内 mention 语义自动生效(watch/serve/webhook 全部照常唤醒)。

6.4 重构:抽 postMessageAsIdentity

现 POST /api/channels/:slug/messages(worker/src/index.ts:2855)的"构造 identity 头 → fetchChannelDO"逻辑抽成可直接调用的函数,路由与桥接共用。桥接侧构造的身份:{name: handle 或 display_name, kind: "human", role: "human", account}——kind 必须是 human,这样桥接消息天然是 loop guard 的人类锚点。

6.5 M2 验收标准

  1. 后台"保存请求地址"探活通过(challenge 握手);
  2. 群里 @bot bind dev(发送者是 dev 的 moderator)后,@bot 帮我看看 CI @claude-code 出现在 dev 频道,署名为该人类,且频道里的 claude-code 被正常唤醒;
  3. 非 moderator bind 被拒并有群内回执;未登录过的 Lark 用户发言收到引导回执;
  4. Lark 重推同一事件不产生重复消息(dedup 生效);
  5. bun run check 全绿。

10. 风险与坑(提前知道少踩)

坑应对
Lark 发消息限频(单应用约 QPS 50,单用户更低)M1 只投 @mention,量小;relay 失败回 502 即挂上现有重试退避,不需要自建限流
发版要企业管理员审核,周期不可控权限/事件订阅改动一次配齐(M1+M2 的权限一起申请),少走一轮审核
handle 改名后 webhook name 失配(mentions filter 按 name 匹配)lark_notify_subscriptions.target_name 记录注册时的 handle;PUT /api/me/handle 成功后追加一步:遍历本账号订阅,重注册 webhook(M1 先文档标注已知限制,M1.5 补自动迁移)
union_id 为空的老账号D4 email 兜底;都没有则 lark-notify 返回 400 并提示重新登录一次
事件解密的 AES 实现写错(常见错:key 没做 SHA256、IV 取错)照 §6.2 步骤 1 写,单测用 Lark 文档样例密文断言
3 秒超时导致 Lark 重推D6 先 200 后 waitUntil + dedup 表,双保险

11. 小白上手指南(从 clone 到 M1 跑通)

  1. 环境:装 bun;仓库根目录 bun install;跑一次 bun run check 确认基线全绿。
  2. 读代码(半天):按 §2 表格把每个"位置"打开读一遍,重点 worker/src/do.ts 的 dispatchWebhooks→deliverWebhook 链路和 worker/src/index.ts 的 auth callback。
  3. Lark 后台(需要 leo 给协作者权限):登录应用控制台 → 添加"机器人"能力 → 权限管理申请 im:message、im:message:send_as_bot → 创建版本 → 申请发布(企业管理员审核)。M1 只需要这些;M2 的事件订阅配置见 §6.1,建议同一版本一起配。
  4. 写代码:按 §8 顺序:migration → lark.ts(先写纯函数 buildMentionCard + 单测)→ relay 路由 + 单测 → lark-notify 路由 + 单测 → CLI 命令。每步跑 cd worker && bunx vitest run。
  5. 部署验证:cd worker && bun run deploy:xdream(脚本会自动跑 D1 migration);然后按 §5.6 验收。注意:发布前先在频道里发一条决策消息引用本设计(party 礼仪第 7 条,对外动作要有据)。
  6. 提交:commit 用中文 conventional commits(仓库惯例),PR 关联 #62。

工作量参考:M1 约 2~3 人日(含测试),M2 约 3~5 人日(解密与事件联调占大头)。
