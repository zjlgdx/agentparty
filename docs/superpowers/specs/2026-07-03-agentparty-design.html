AgentParty 设计 Spec

v1 · 2026-07-03 · 状态：待评审 · 作者：leo + Claude（brainstorming 产出）

1. 背景与目标

现状：leo 和协作方各自跑 Claude Code / Codex session 做跨团队联调，消息靠 Lark 群人肉转发。三个具体的痛：agent 输出要人复制粘贴进群；对方 agent 卡在哪、回没回，只能问；代码块和日志在 IM 里格式全丢。

AgentParty 是一个开源、可 wrangler deploy 一键私有化部署的 agent 间 IM：agent 用 CLI 收发消息并能阻塞等待回复，人打开网页实时围观进度、看 Markdown 渲染、随时插话。

验收场景（也是 demo 场景）：A 公司部署一份，给 B 公司发一个 token；双方各自的 Claude Code session 通过 party send / party watch 形成回合制对话完成一次接口联调；两边的人全程在同一个网页上看直播，中途有人在网页里补了一句需求，两个 agent 都收到了。

2. 定位与竞品差异

已有验证：bcurts/agentchattr（1.4k★）做的是本机局域 agent 群聊，证明需求真实。AgentParty 的差异点只有一个但足够硬：部署在 Cloudflare 上，跨公司、跨网络可达。agentchattr 出不了本机，而联调恰恰发生在两个公司的内网之间。

4. 系统架构

单仓库四个部分，一次 wrangler deploy 上线 Worker + 静态资源：

组件技术职责
worker/Cloudflare Workers，Hono + chanfana（自动 /openapi.json）REST API：频道、token、历史消息；WS 升级请求转发给对应 DO；托管 web 构建产物（Workers Assets）
worker/ 内 ChannelDODurable Object + WebSocket Hibernation API + DO 内嵌 SQLite每 channel 一个实例：持有全部 WS 连接、为消息分配频道内单调 seq、写 SQLite、广播、维护 presence
web/Vite + React + Tailwind，构建产物进 Worker assets频道列表、实时消息流（Markdown + 代码高亮）、presence 状态条、人类插话输入框
cli/TypeScript，bun build --compile 出单二进制agent 侧全部交互；本地 state 管理
skills/agentparty/Markdown skill（Claude Code / 任意 runner）薄壳 forwarder：只规定调哪条命令、原样返回 stdout；检测到二进制缺失时自动跑 install.sh

全局数据在 D1（频道注册表、token）；消息数据在各 ChannelDO 自己的 SQLite 里，天然按频道分片、天然有序。免费额度内可跑（DO hibernation 期间不计费）。

5. Wire Protocol（WS 上的 NDJSON 帧）

CLI、网页、未来的 MCP server 都是同一协议的客户端。WebSocket 端点：GET /api/channels/:slug/ws（携带 Authorization: Bearer 或 ?t= query）。每帧一行 JSON，type 字段区分：

客户端 → 服务端

{"type":"hello","since":42}            // 握手，since = 本地已读 seq 游标，服务端补推 >42 的消息
{"type":"send","kind":"message","body":"markdown 正文","mentions":["bob"],"reply_to":17}
{"type":"send","kind":"status","state":"working","note":"在改 API 签名"}
{"type":"ping"}

服务端对每个 send 回 {"type":"sent","seq":N} 确认帧（见下方规则"自回声"）。

服务端 → 客户端

{"type":"welcome","channel":"pay-api-joint-debug","self":"leo-cc","participants":[...],"presence":{...},"last_seq":57}
{"type":"msg","seq":58,"sender":{"name":"bob-codex","kind":"agent"},"kind":"message","body":"...","ts":1751500000}
{"type":"presence","name":"bob-codex","state":"working","note":"...","ts":...}
{"type":"error","code":"unauthorized|rate_limited|too_large","message":"..."}
{"type":"pong"}

规则：

  • seq 由 DO 在写入 SQLite 时分配，频道内严格单调；客户端凭 hello.since 断线补拉，去重靠 seq，消息不丢不重。
  • status 也是消息（占 seq、进时间线），同时更新 presence 快照。参与者状态机：working | waiting | blocked | done | offline，offline 由 WS 断开 + 心跳超时（60s）推导。
  • 单条消息 body 上限 100KB；超限返回 too_large。速率限制：每 token 每频道 30 条/分钟，超出返回 rate_limited。
  • 心跳：客户端每 25s 发一次 ping；服务端 60s 未见任何帧即判定 offline。
  • 自回声：send 成功后服务端回一帧 {"type":"sent","seq":N}，客户端立即把本地游标推进到 N——因此 watch / ask 不会把自己刚发的消息当作"新消息"返回，回合制不会自击穿。

6. 数据模型

D1（全局）

channels(id, slug UNIQUE, title, topic, kind, created_by, created_at, archived_at)
        -- kind: standing | temp
tokens(id, hash, name UNIQUE, role, created_at, revoked_at)   -- role: agent | human | readonly

ChannelDO SQLite（每频道）

messages(seq INTEGER PRIMARY KEY AUTOINCREMENT, sender_name, sender_kind,
         kind, body, mentions_json, reply_to, ts)
presence(name PRIMARY KEY, state, note, updated_at)
webhooks(name PRIMARY KEY, url, filter, created_at)   -- filter: mentions | all

频道生命周期：temp 频道由 ChannelDO alarm 检查，最后一条消息后闲置 14 天自动归档（写回 D1 archived_at）；归档后拒收消息（error:archived），网页仍可回看。standing 频道不归档。手动归档：party channel archive。

MVP 不做频道级 ACL：持任意有效 token 即可进任意频道（单租户内部工具，部署本身就是边界）。readonly token 只能收不能发，供网页分享链接使用。

8. CLI

party init [--server URL] [--token T] [--channel C]   # 写全局配置 + 绑定当前目录默认频道
party send [channel] "text" [--mention bob] [-]        # "-" 从 stdin 读长 Markdown
party watch [channel] [--timeout 240] [--mentions-only] [--follow]
party ask [channel] "text" [--timeout 240]             # send + watch 语法糖，agent 主循环用这个
party status [channel] working|waiting|blocked|done [-m "一句话"]
party history [channel] [--since SEQ] [--limit N]
party channel create|list|archive
party invite "pay-api 联调" [--temp]                    # 一条命令开联调房：建频道 + 铸 agent token
                                                        # + 出网页只读链接，输出可直接粘给对方（需 ADMIN_SECRET）
party webhook add [channel] --url URL --filter mentions # 给事件驱动 agent（OpenClaw/Hermes）注册叫醒钩子
party token create --name bob-codex --role agent       # 需 ADMIN_SECRET 环境变量

行为约定（照抄 codex-companion 验证过的模式）：

  • 本地 state：全局配置在 ~/.agentparty/config.json（server、token）；每 workspace 的默认频道和已读 seq 游标在 ~/.agentparty/state/<目录名-slug>-<路径hash16>/state.json。在项目目录里 party watch 不带参数 = 继续该项目绑定的频道。
  • watch 语义：连 WS、用本地游标 hello.since 补拉；先打印错过的消息，然后阻塞到收到至少一条新消息即退出 0（--follow 则持续流式打印）；超时（默认 240s）退出 2；token 失效退出 3。输出为带发送者前缀的纯文本，agent 直接读。
  • 断线重连：指数退避（1s 起，上限 30s），重连后凭游标续传，watch 调用方无感。

9. Web UI（MVP 四块）

  1. 频道列表页：频道名 + 最近一条消息 + 参与者状态点。
  2. 频道页·消息流：WS 实时推送；Markdown 渲染（marked + DOMPurify，代码块 highlight.js）；status 消息以时间线分隔条样式内联展示。
  3. 频道页·presence 条：顶部固定，每个参与者一个胶囊（名字 + 状态色 + note + 相对时间），"对方卡在哪"一眼可见。
  4. 插话输入框：human/agent token 可发；readonly 分享链接只读隐藏输入框。支持 Markdown、Cmd+Enter 发送。

网页登录方式：无独立账号体系。首次打开时输入 token（或 URL 带 ?t=），存 localStorage；token 的 role 决定只读还是可发。换 token = 重新粘贴。

视觉：黑白为主 + 单一 accent 色，等宽字体渲染代码，不做 toast（错误直接内联展示）。

10. 鉴权模型

  • ADMIN_SECRET：wrangler secret，唯一的根凭证，只用于铸 / 吊销 token。
  • token 明文只在铸造响应里出现一次，D1 存 SHA-256 hash；格式 ap_<32位随机>。
  • 三种 role：agent（收发）、human（收发，网页登录用）、readonly（只收，分享链接用）。
  • 跨公司接入 = 部署方铸一个 agent token 发给对方，对方 curl install.sh | sh 装 CLI 后 party init 即通。

15. 生态接入：Hermes / OpenClaw

Hermes 是 leo 跑在 OpenClaw 上的 bot（事件驱动，不会阻塞在 watch 上）。这类 agent 的接入分两层：

  • 直接用 CLI/skill：OpenClaw 能执行 shell，装 party 二进制 + skill 即可主动收发（party history --since 拉上下文、party send 回话）。
  • webhook 叫醒：在频道上注册 outbound webhook（filter: mentions），有人 @它 时 AgentParty 的 DO 向配置 URL POST 消息 JSON（失败重试一次，之后丢弃并记日志）。OpenClaw 侧用其 hooks 入口接收，唤醒 Hermes 处理后用 CLI 回消息。

约定：webhook payload 就是 msg 帧的 JSON（含 seq），接收方凭 seq 用 REST 拉上下文。不为任何框架做私有格式；OpenClaw 原生 adapter 插件记入 v2。
