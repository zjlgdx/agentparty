AgentParty 桌面端成熟化实施计划

2026-07-10 · 基于 2026-07-09-desktop-tauri-design.html · Tauri v2 / React / GitHub Releases

目标与边界

把当前“可打开 web UI 的 macOS 壳”推进为可长期安装使用的桌面应用。成熟应用的最低闭环是：版本可信、可发现并安装更新、后台可接收提醒、登录不依赖复制敏感 token、崩溃或网络失败有可理解状态、发布产物可验证。

发布仍走 GitHub Releases，不进入 npm registry 或应用商店。桌面版本与 CLI 共用 vX.Y.Z 标签；应用内更新使用 Tauri 强制签名的 updater bundle，DMG 继续作为首次安装入口。

能力当前证据判定
Tauri 壳与远端 APIdesktop/、运行时 API base、Tauri CORS、v0.2.82 双架构 DMG已完成 M1

系统通知与托盘只有浏览器 Notification；关窗即退出缺失
桌面登录只能粘贴 token不适合普通用户
签名/公证DMG 无 Developer ID 签名和公证发布风险
Windows/Linux设计中有方向，CI 只构建 macOS后续里程碑

里程碑

顺序交付验收门
R1可靠更新：单一版本源、签名 updater bundle、静态更新清单、检查/下载/安装/重启 UI旧 DMG 安装版可升级到下一测试 tag
R2系统驻留：Tauri 通知、托盘、关窗隐藏、未读角标、开机自启开关后台收到 @ 后通知；点击聚焦频道；托盘可真正退出
R3配对登录：网页签发一次性短码，桌面兑换独立 session；deep link 仅作加速10 秒内完成登录；短码一次性、5 分钟过期、失败限流
R4本机 agent：受控启动/停止 party serve sidecar，日志、状态与失败恢复用户无需终端即可把本机 agent 置为可唤醒
R5分发硬化：Developer ID 签名、公证、Windows NSIS、Linux AppImage、跨平台升级 smoke干净机器下载即开；每个平台完成 N-1 → N 更新

R1 文件结构

文件责任
scripts/release-version.ts + test读取/校验/写入 CLI 与 desktop 的唯一发布版本；release 脚本只调用它
scripts/desktop-update-manifest.ts + test从两个架构的 updater bundle 和签名生成 Tauri 静态 latest.json
web/src/lib/desktopUpdater.ts + test隔离 Tauri 动态 import、更新状态机、进度归一化和错误分类
web/src/components/DesktopUpdater.tsx仅桌面显示的更新入口和状态浮层；手动检查、安装并重启
web/src/i18n/strings/DesktopUpdater.ts中英文更新文案
desktop/src-tauri/*注册 updater/process 插件、权限、公钥、endpoint、生成 updater artifacts
.github/workflows/release.yml注入签名密钥、重命名架构包、生成并上传 latest.json

R1 执行步骤

任务 1：版本源

  1. 先写测试：CLI/desktop 版本不一致时失败；写入后两个 package 同步；非法 semver 不写文件。
  2. 实现 release-version.ts，让 tauri.conf.json 的 version 指向 ../package.json。
  3. 修改 scripts/release.sh，同时 bump CLI 与 desktop，并在失败时恢复两份文件。
  4. 门禁：bun test scripts/release-version.test.ts 与 bun run desktop:typecheck。

任务 2：更新清单

  1. 先写测试：两架构齐全时生成正确平台键和下载 URL；缺 bundle、缺签名、空签名都失败。
  2. 实现纯函数和 CLI 入口，输出 RFC 3339 pub_date、版本、notes、darwin-aarch64/darwin-x86_64。
  3. Release job 在上传前生成清单，清单与 bundle 一起成为 release asset。

任务 3：应用更新状态机

  1. 先写 reducer/服务测试：无更新、发现更新、下载进度、安装完成、网络错误、重复点击 single-flight。
  2. 动态加载 @tauri-apps/plugin-updater 和 @tauri-apps/plugin-process，浏览器构建不调用插件。
  3. 首次启动延迟检查一次；之后 24 小时内不重复自动检查；手动检查永远可用。
  4. 安装必须由用户确认；下载展示百分比或字节进度；安装完成后调用 relaunch。

任务 4：Tauri 与发布签名

  1. 添加 Rust/JS 插件，注册 updater:default 与 process:allow-restart 权限。
  2. 配置 HTTPS endpoint、不可提交的私钥、可公开的 pubkey、createUpdaterArtifacts: true。
  3. GitHub Actions 从 TAURI_SIGNING_PRIVATE_KEY / TAURI_SIGNING_PRIVATE_KEY_PASSWORD 读取密钥；任何一个 updater 产物缺失即失败。
  4. 保留 DMG 与 sha256；新增每架构 .app.tar.gz、.sig 和 latest.json。

错误与安全

场景行为
离线/超时不阻塞应用；更新入口显示可重试错误，自动检查不弹打断式对话框
签名不匹配安装失败且不重启；保留当前版本；显示“更新校验失败”
下载中重复点击复用同一 Promise，不创建第二次下载
密钥泄露/丢失私钥不进入 git；GitHub secret + 用户控制的离线备份；丢失视为不可恢复的发布事故
Web 浏览器不渲染更新入口，不 import 原生插件，不改变现有登录和频道行为

R1 验收命令

bun test scripts/release-version.test.ts scripts/desktop-update-manifest.test.ts
cd web && bun test && bunx tsc --noEmit && bunx vite build
cd desktop && bun run typecheck
TAURI_SIGNING_PRIVATE_KEY=... bun run desktop:build:prod
bun run check

最终证据不是“workflow 绿”，而是用旧版 DMG 安装应用，再发布一个更高版本的测试 tag，应用内发现更新、下载、验签、替换并重启，重启后显示新版本。
